# Strategy Composition and Execution Rules

## 역할

이 문서는 매수전략을 완성하기 전에 전략 간 충돌, 중복 신호, 알림, 실제 주문 연결을 어떻게 나눌지 정한 기준서다.

전략이 같은 종목에서 동시에 신호를 내더라도, 전략 클래스가 직접 매수/알림 여부를 결정하지 않는다. 전략 클래스는 신호와 근거만 반환하고, 실행 여부는 전략실 키와 주문/리스크 레이어가 결정한다.

## 레이어 분리

```text
전략 슬롯
→ 중복 정책
→ Engine Start / Live Orders
→ RiskGuard
→ Telegram / Order
```

각 레이어 역할:

- 전략 슬롯: 조건 평가, 점수, 단계, 근거, 후보 신호만 만든다.
- 중복 정책: 같은 종목에 여러 전략이 걸렸을 때 알림/매수 허용 여부를 결정한다.
- Engine Start: 전략 평가와 기록/알림 엔진을 켜는 스위치다.
- Live Orders: 실제 주문 레이어로 넘길지 결정한다.
- RiskGuard: 예산, 슬롯, 보유, 당일 재매수, 주문 가능 수량, 손절/목표가를 확인한다.
- Telegram: 알림 허용 정책에 따라 메시지를 보낸다.
- Order: 모든 게이트를 통과한 경우에만 주문을 보낸다.

## 단정 금지

중복 전략 발생 시 "한 번만 알림/매수"라고 단정하지 않는다.

처리는 항상 전략실 키 옵션을 따른다.

## 중복 정책 기본 매트릭스

```text
중복매수 ON  + 중복알림 ON  = 전략별 알림/매수 가능
중복매수 OFF + 중복알림 ON  = 매수는 대표 1회, 알림은 전략별 가능
중복매수 ON  + 중복알림 OFF = 매수는 전략별 가능, 알림은 대표 1회
중복매수 OFF + 중복알림 OFF = 대표 전략 1개만 알림/매수
```

기본값은 보수적으로 본다.

```text
중복매수 OFF
중복알림 ON
```

즉, 같은 종목에 여러 전략이 겹치면 알림은 전략별로 볼 수 있지만 실제 매수는 기본적으로 한 번만 허용한다.

## Engine / Live Orders 기준

```text
Engine Start OFF = 전략 평가 없음
Engine Start ON + Live Orders OFF = 알림/기록만, 실주문 없음
Engine Start ON + Live Orders ON = 주문 후보를 RiskGuard로 전달
```

Live Orders가 OFF라도 전략 평가는 가능하다. 이 경우 전략 검증, 알림, 로그, 진행상황 표시만 수행한다.

## 대표 전략 선정

중복알림 또는 중복매수가 꺼져 있을 때 대표 전략이 필요하다.

대표 전략은 아래 우선순위로 고른다.

1. 사용자가 수동으로 우선순위를 지정한 전략
2. 전략 점수 또는 진행률이 높은 전략
3. 안정형 전략
4. 중간형 전략
5. 공격형 전략
6. 보조 전략

단, 주 매매 전략과 보조 전략이 동시에 걸리면 보조 전략은 단독 대표가 될 수 없다. 보조 전략은 주 전략의 설명/점수 보강 역할만 한다.

## 전략 진행률 해석

전략 진행률 100%를 매수 조건 충족으로만 쓰지 않는다.

전략 진행률은 검색식 편입부터 매수 후보, 실제 매수, 매도 추적, 종료까지 포함한 전체 사이클로 본다.

기본 배분은 아래 둘 중 하나를 우선 검토한다.

```text
매수 준비/진입: 60%
매도 추적/종료: 40%
```

또는:

```text
매수 준비/진입: 70%
매도 추적/종료: 30%
```

즉, 진행률이 60~70%에 도달하면 매수 후보 또는 매수 실행 구간으로 보고, 나머지 30~40%는 보유 후 매도 추적에 사용한다.

100%는 매수 시점이 아니라 매도까지 끝난 전략 종료 상태다.

예시:

```text
10%  검색식 편입
20%  기본 수치/시장 확인
30%  거래대금/등락률 확인
40%  눌림 감지
50%  회복 확인
60%  돌파 대기 또는 매수 후보
70%  매수 실행/보유 진입
80%  1차 목표 또는 손절선 추적
90%  익절/트레일링/위험 관리
100% 매도 완료, 전략 종료
```

전략별로 매수 기준점은 다를 수 있지만, 매수 이후 매도 추적 구간을 반드시 남긴다.

## 주전략/보조전략/후보패턴 구분

```text
주전략 = 매수 후보 신호를 만들 수 있음
보조전략 = 뉴스/공시/테마/위험 태그 보강, 단독 매수 신호 없음
후보패턴 = 문서 보관 또는 알림 검증용, 주문 연결 없음
```

후보패턴은 실전 전략 슬롯 UI에 올리기 전까지 주문 레이어로 넘기지 않는다.

## SOR 당일강세 눌림 돌파 계열

아래 전략들은 같은 계열로 본다.

```text
15분봉 60선 눌림 + 5분봉 20봉 신고가 = 안정형
10분봉 60선 눌림 + 5분봉 20봉 신고가 = 중간형
10분봉 60선 눌림 + 3분봉 20봉 신고가 = 공격형
```

같은 종목에서 이 계열 신호가 여러 개 발생하면 중복 정책을 따른다.

공격형이 먼저 발생하고 중간형/안정형이 나중에 발생하면, 중복매수가 OFF일 때는 후속 신호를 강화 기록으로만 남긴다.

## 기존 조건00 MA 회복초입과의 관계

기존 프로그램의 `조건00 MA 회복초입`은 아래 구조다.

```text
KRX 일봉 기준봉
→ 기준봉 이후 10분 MA60 이탈
→ 10분 MA60 회복
→ 10분 MA5 > MA20 / MA60
→ 5분 20봉 신고가
```

이 전략은 SOR 당일강세 눌림 돌파 계열과 일부 조건이 겹치지만 동일 전략으로 합치지 않는다.

중복 발생 시 하나를 자동으로 제거하지 말고, 중복 정책과 대표 전략 선정 규칙을 따른다.

## 주문 안전

- 전략 클래스에서 직접 주문을 보내지 않는다.
- 전략 클래스에서 주문 수량을 확정하지 않는다.
- `_NX`, `_AL` 같은 시장 데이터 suffix를 주문 종목코드에 넣지 않는다.
- SOR 주문은 주문 레이어에서 `dmst_stex_tp = SOR`로 처리한다.
- NXT/SOR 주문은 시장가를 기본값으로 단정하지 않는다.
- 주문은 항상 RiskGuard 통과 후 실행한다.

## 문서/코드 반영 원칙

새 전략을 추가할 때는 아래 순서를 지킨다.

1. 전략 설명서 작성
2. 전략 슬롯 클래스 추가
3. registry 등록
4. UI 토글 연결
5. 진행상황 표시 연결
6. 중복 정책 연결
7. 알림/주문 레이어 연결
8. 빌드와 로그 검증

이 순서를 건너뛰고 주문 연결부터 하지 않는다.
