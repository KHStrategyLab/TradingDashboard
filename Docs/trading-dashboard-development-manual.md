# TradingDashboard 개발 매뉴얼

작성일: 2026-05-29

이 문서는 TradingDashboard를 계속 개발할 때 흔들리지 않기 위한 기준 문서다. 화면의 각 영역이 어떤 REST TR, WebSocket 실시간 항목, 캐시, 계산식으로 채워지는지 기록한다. 기능을 추가하거나 수정할 때는 이 문서를 먼저 보고, 새 화면 항목이 생기면 데이터 출처와 갱신 규칙도 같이 업데이트한다.

## 1차 목표

이번 프로그램의 1차 목표는 호가창 완성이다.

- 키움/MTS에서 보는 호가, 체결, 거래량, 거래대금, 등락률, 기준가 관련 수치와 화면 값이 일치해야 한다.
- 매수/매도 호가, 잔량, 현재가, 체결량, 매수체결량, 매도체결량은 실시간과 REST 스냅샷을 조합해 MTS 기준으로 검증한다.
- KRX/NXT 시간대가 달라도 기준가와 색상 기준은 흔들리지 않아야 한다.
- 장중에는 WebSocket 실시간 값을 우선하고, 장마감 이후에는 REST 최종 스냅샷으로 빈 화면 없이 복원한다.
- 수치가 MTS와 다르면 UI 모양보다 데이터 출처, TR 필드, 계산식을 먼저 의심한다.
- 호가창과 오른쪽 체결/종목정보 수치가 안정된 뒤에 추적기, 검색식 확장, 매매 판단 보조 기능을 붙인다.

## 1차 달성 기록

2026-05-29 기준, 하루 작업으로 1차 목표의 핵심 골격을 완성했다.

- `Kiwoom.ConditionSeq01` 설정값에 해당하는 조건검색식에서 종목 목록을 받아 왼쪽 목록에 표시한다.
- 조건검색 실시간 추적으로 장중 편입/이탈을 처리한다.
- 선택 종목의 KRX 전일종가를 기준가로 잠그고, 화면 색상 기준으로 사용한다.
- 선택 종목의 호가창을 `0D/0H` 실시간과 REST 스냅샷으로 구성한다.
- 관심종목 전체의 `0B` 체결 실시간을 받아 현재가, 거래량, 체결 정보를 움직인다.
- 오른쪽 체결/종목정보 영역에 MTS와 맞춰야 할 핵심 수치를 배치했다.
- 일봉/주봉/월봉/분봉 차트를 붙이고 0B 실시간 현재가를 차트에 반영한다.
- KRX/NXT 구분, 장마감 후 스냅샷, 기준가 덮어쓰기 금지, 캐시 복원 규칙을 정리했다.

아직 세부 수치 검증과 TR 보정은 남아 있지만, 프로그램의 뼈대는 완성 단계에 들어왔다. 이후 개발은 "MTS와 수치 일치"를 최우선 기준으로 검증하고 보정한다.

## 개발 분기 기준점

2026-05-29의 현재 버전은 이후 다른 개발을 시작할 때 사용할 중간 기준점이다.

이 기준점의 의미:

- 호가창, 실시간 체결, 차트, 뉴스, 공시, 조건검색 추적이 한 화면 안에서 연결된 상태다.

- 새 기능 개발 중 문제가 커지면 이 기준점으로 돌아와 다시 분기할 수 있다.

## 전략분봉 데이터관리부

전략분봉 데이터관리부의 주인은 `StrategyMinuteCacheService`다.

이 서비스는 화면 차트가 아니라 전략실 숫자 장부다. 종목별, 시장별, 분봉별로 확정봉을 저장한다.

키 형식:

```text
종목코드|시장|분봉
예: 018260|KRX|5
예: 018260|NXT|10
```

저장 단위:

- 종목코드
- 시장: KRX/NXT
- 분봉: 3/5/10/15 등
- 봉 시간
- 시가
- 고가
- 저가
- 종가
- 거래량
- 거래대금
- MA5/MA10/MA20/MA60/MA200/MA240/MA480

운영 원칙:

- 전략은 원본 봉 리스트를 직접 뒤지지 않는다.
- 전략은 `StrategyMinuteFrameSnapshot` / `StrategyMinuteSnapshotSet`으로 정해진 숫자 묶음만 받는다.
- 봉마감 시 `ApplyClosedBar(...)`로 확정봉의 시고저종/거래량/거래대금을 받아쓴다.
- 확정봉이 들어오면 같은 시간 봉은 교체하고, 새 봉은 추가한다.
- 확정봉 저장 직후 이평선을 다시 계산해서 해당 봉에 고정한다.
- 현재 진행봉은 참고값일 수 있으나, 매수/매도 확정 조건은 기본적으로 마감된 확정봉 기준이다.
- 화면 차트 `_currentChartCandles`는 표시용이며, 전략 장부의 소유자가 아니다.
- 분봉 파일 캐시는 사용자가 `파일 저장`을 켠 경우에만 seed 재사용 용도로 사용한다. 장중 판단은 파일이 아니라 메모리 전략 장부 기준이다.

현재 1차 연결 상태:

- `LoadStrategyMinuteDataAsync(...)`는 1/3/5/10/15/30분봉을 받아 `StrategyMinuteCacheService`에 seed로 넣는 내부 로더다.
- 1분봉은 향후 단기 전략/이평선 확인용으로 최소 300개를 seed로 받는다.
- 전략실의 `분봉 프리로드` 스위치가 ON이면 조건식/캐시 감시목록 적용 후 idle 예약을 걸고, 사람이 종목을 하나씩 선택하지 않아도 감시목록 분봉 seed를 자동 로드한다.
- idle 대기 시간은 전략실의 `대기시간` 입력칸 또는 `Config/local.settings.json`의 `StrategyMinutePreload.IdleDelaySeconds`로 조절한다. 입력칸 변경값은 `local.settings.json`에 저장되며, 기본값은 180초다.
- `파일 저장` 스위치와 `대기시간` 입력칸은 `분봉 프리로드`를 켜기 전에 정한다.
- `분봉 프리로드`가 ON이면 `대기시간` 입력칸과 `파일 저장` 스위치를 잠그고, `분봉 프리로드` 스위치는 중단용으로 남겨둔다.
- idle이 끝나 프리로드가 시작되면 종목 사이에 추가 대기 없이 연속 실행한다. 새 조건 편입이나 스위치 변경이 들어오면 진행 중 배치를 끊고 새 idle 예약으로 다시 잡는다.
- 장중 조건식 신규 편입 종목도 감시목록에 들어오면 idle 예약에 포함한다.
- 전략실의 `파일 저장` 스위치가 ON이면 `Storage/StrategyMinuteSeeds/{yyyyMMdd}/{code}_{market}_{minute}.json`에 분봉 seed를 저장한다.
- 다음 프리로드 때 같은 날짜/종목/시장/분봉 파일이 있으면 파일을 먼저 읽고, 목표 개수가 부족한 경우에만 REST로 보충한 뒤 다시 저장한다.
- 선택 종목 프리로드와 자동 프리로드가 겹치면 자동 프리로드는 해당 종목 완료를 기다린 뒤 READY/파일 저장 상태에 포함한다.
- `StrategyMinuteDataStatus`는 차트 메모리 캐시가 아니라 전략분봉 장부의 READY/개수 상태를 표시한다.
- `StrategyMinuteSnapshotSet`은 전략 슬롯이 요청할 수 있는 분봉별 Snapshot 묶음이다.
- 0B 실시간 틱은 `StrategyMinuteCacheService.ApplyRealtimeTick(...)`으로 전략 장부의 현재봉을 갱신한다.
- 0B 거래량은 누적거래량 `13`의 직전값 차분을 우선하고, 차분을 만들 수 없을 때 체결량 `15`를 보조로 사용한다.
- 0B 연결은 숫자 장부 갱신까지만 담당하며, 매수신호나 주문을 직접 만들지 않는다.
- 테스트 매매는 `Engine Start ON + Live Orders OFF + Paper Trading ON` 조합으로 사용한다. 분봉 장부 READY 전에는 Paper BUY 로그도 내지 않는다.
- 봉마감 확정봉 입력구는 `ApplyClosedBar(...)`로 준비되어 있다.

전략별 기억값은 공통 장부에 넣지 않는다.

예:

- 첫 5분봉 20봉 신고가 봉의 시고저종
- 첫 신고가 봉 거래량/거래대금
- 마디 시작가/고가/저가
- 마디 거래량/거래대금/효율
- 눌림/재돌파/매수완료/매도추적 단계

이런 값은 향후 `StrategyRuntimeState` 같은 전략별 상태 저장소가 가진다. 공통 장부는 객관적인 봉 수치만 제공한다.

## 핵심 원칙

- 기준가는 항상 KRX 전일종가다.
- 기준가는 NXT 종가, 현재가, LastPrice, 호가, 차트 종가, 캐시의 임의 가격으로 덮어쓰면 안 된다.
- NXT 종목을 보고 있어도 색상 판단과 등락률 기준은 KRX 전일종가다.
- 화면의 가격 색상은 기준가 초과 빨강, 기준가 미만 파랑, 기준가 동일 흰색이다.
- 조건검색식은 주기적 리로드가 아니라 WebSocket 조건검색 실시간 추적으로 편입과 이탈을 받는다.
- 한번 들어온 종목의 기본정보는 캐시에 남긴다. 이탈했다가 재편입되면 REST를 다시 긁지 않고 캐시로 복원하고 0B만 다시 맞춘다.
- 종목 선택이 바뀌면 이전 선택의 늦은 응답은 화면에 반영하지 않는다.
- 빈 응답이나 신뢰할 수 없는 값으로 기존의 유효한 화면 값을 지우지 않는다.

## 왼쪽 관심종목 목록

화면 위치:

- 왼쪽 `최근 조회 / 관심종목`

초기 조회:

- WebSocket `CNSRLST`: 조건검색식 목록 조회
- WebSocket `CNSRREQ`
- `search_type = "1"`
- `stex_tp = "K"`

`search_type = "1"`은 단순 조회가 아니라 조건검색 실시간 추적 등록이다. 최초 결과를 받은 뒤 장중 편입/이탈 이벤트가 같은 WebSocket으로 들어온다.

장중 편입 처리는 폴링하지 않는다. 조건검색 WebSocket이 던지는 `REAL` 편입 이벤트만 받아 처리한다. 신규 편입 종목은 순수 관찰 후보이므로 `NEW` 배지만 붙이고, 그 순간에는 일봉 게이트를 다시 조회하지 않는다. 오늘 캐시에 게이트 통과 이력이 이미 있는 종목만 `D+`로 복원한다. 편입 이벤트가 몰릴 때도 기본정보/뉴스/공시 보강은 순차 처리하고, 같은 종목이 이미 대기 중이면 중복 작업을 만들지 않는다.

편입 보강 중 이탈 이벤트가 먼저 도착할 수 있으므로 코드별 최신 세대번호를 저장한다. 최종 목록 삽입 직전에 해당 코드가 아직 최신 enter 상태인지 확인하며, exit가 더 최신이면 add를 중단한다. `enter -> exit -> enter`가 빠르게 반복되면 마지막 enter만 유효하게 처리한다.

초기 구성 순서:

1. `CNSRLST`로 조건식 목록 조회
2. `Kiwoom.ConditionSeq01`에 해당하는 seq 확인

## NXT 종목의 표시 기준

MTS와 맞춘 중요한 규칙이다.

2026-05-29 기준, 삼성에스디에스 등 NXT 가능 종목을 MTS SOR ON 화면과 대조해 거의 일치하는 것으로 확인했다. 이 항목은 기준수치 확정 규칙으로 본다.

- 기준가: KRX 전일종가
- 색상 기준: KRX 전일종가
- 등락률 기준: KRX 전일종가 대비
- NXT 시간대의 시가/고가/저가/현재가: NXT 시세
- NXT 시간대의 호가/잔량: NXT 호가
- NXT 시간대의 거래량/거래대금: MTS SOR ON 기준에 맞춰 `_AL` 통합 일별거래상세를 우선 표시

즉, NXT 종목이라고 해서 기준가까지 NXT 20시 종가로 바꾸면 안 된다. 반대로 NXT 시간대에 보고 있는 시가/고가/저가/현재가를 KRX 값으로 강제로 되돌려도 MTS와 어긋난다.

화면 예:

- `기준가`: KRX 전일종가
- `시가`, `고가`, `저가`: 현재 선택 시장이 NXT이면 NXT 값
- 가격 글자색: 위 시가/고가/저가 값이 KRX 전일종가보다 높은지 낮은지로 판단

구현 기준:

- `GetStockStatusMetricsByGuideAsync(code, useNxtMarket: true)`는 `_NX` 코드로 OHLC/현재가 계열을 조회한다.
- `InfoBasePriceText`는 `StockStatusMetrics.BasePriceText`를 쓰지 않고, 선택 시작부에서 잠근 `_krxPrevClosePrice`만 표시한다.
- `ApplySelectedPriceInfoColors`는 NXT OHLC 가격을 KRX 전일종가와 비교해서 색상을 정한다.
- NXT 장중에는 KRX 종가 스냅샷을 적용하지 않는다. 장중 스냅샷이 들어오면 NXT 시/고/저/현재가를 KRX 값으로 덮어 MTS SOR ON 화면과 어긋난다.
- NXT 장중/고정 시간의 거래량, 거래대금, 전일거래량비율은 `_AL` 통합 일별거래상세로 보정하고, 이후 NXT 0B 단일 거래량이 이 값을 덮지 않도록 유지한다.
- NXT/SOR 표시 중 NXT 조회값이 비어도 KRX 시가/고가/저가/종가/호가/체결량으로 재조회해 채우지 않는다. KRX에서 가져와도 되는 값은 기준가로 잠근 전일종가뿐이다.

확정 상태:

- MTS SOR ON 기준으로 `기준가`, `시가`, `고가`, `저가`, `종가`, `거래량`, `거래대금`, 호가 색상 기준이 실사용 가능한 수준으로 맞았다.
- 약간의 차이는 REST 응답 시점, MTS 갱신 시점, NXT/통합 집계 반영 지연으로 보고 항목별 미세 보정 대상으로 둔다.
- 새 기능에서 이 규칙을 건드릴 때는 먼저 MTS 대조 로그를 남기고 수정한다.

## 수치 소유권 꼬리표 원칙

화면의 각 수치는 전담 데이터 출처를 가진다. 다른 TR이나 실시간 이벤트가 늦게 도착했다는 이유로 전담자가 아닌 값이 칸을 덮으면 안 된다.

이 원칙은 프로그램 칸에서 확인되었다. `ka90008`로 프로그램 순매수금액을 정상 표시했지만, 이후 `0B` 실시간 체결 갱신이 `_currentStatusMetrics`를 다시 그리면서 프로그램 값을 `-`로 덮었다. `0B`에는 프로그램매매 값이 없으므로 이 칸을 갱신할 권한이 없다.

소유권 예:

- 기준가: KRX 전일종가. NXT, `_AL`, `0B`가 덮을 수 없다.
- 시가/고가/저가/현재가: 현재 선택 시장의 시세 TR과 해당 시장 실시간.
- 거래량/거래대금: `_AL` 통합 일별거래상세 우선. `0B`는 보조로만 사용한다.
- 전일거래량대비: `_AL` 통합 일별거래상세와 전일 거래량.
- 매수/매도 체결량: 체결 요약 TR이 초기값을 소유하고, `0B`는 증가분 보조.
- 호가: `0D`와 호가 REST 스냅샷.
- 최근 체결: `0B`.
- 프로그램: `ka90008`. `0B`가 공란으로 덮을 수 없다.

구현 기준:

- 수치를 화면에만 쓰지 말고 해당 수치의 상태 객체에도 저장한다.
- 실시간 갱신은 자기 소유 칸만 바꾼다.
- 새 칸을 추가할 때는 먼저 "owner TR/WS", "보조 소스", "덮기 금지 소스"를 정한다.
- `ka90008` 프로그램 값은 `_currentStatusMetrics`에도 저장해 `0B` 실시간 재표시가 공란으로 덮지 못하게 한다.

## 0s 장운영구분

용도:

- KRX/NXT 사용 여부 판단
- NXT 장후 등록 여부 판단
- KRX 장마감 후 REST 스냅샷 보강 여부 판단

등록:

```json
{
  "trnm": "REG",
  "grp_no": "899",
  "refresh": "1",
  "data": [
    {
      "item": [""],
      "type": ["0s"]
    }
  ]
}
```

주요 필드:

- `215`: 장운영구분 코드
- `214`: 예상잔여시간 보조값
- `20`: 이벤트 시각

NXT 열림 판단:

- `P`, `R`, `T`, `U`

NXT 닫힘 판단:

- `Q`, `S`, `V`

초기 선조회가 시간초과되면 KRX 우선으로 시작하고, 이후 0s 이벤트가 오면 상태를 갱신한다.

```json
  "refresh": "1",
  "data": [
    {
      "item": ["005930"],
      "type": ["0B"]
    }
  ]
}
```

NXT 장후 또는 NXT 가능 시간에는 NXT 가능 종목을 `{종목코드}_NX`로 추가 등록한다.

주요 필드:

- `10`: 현재가
- `11`: 전일대비
- `12`: 등락률
- `13`: 누적거래량
- `15`: 체결량
- `20`: 체결시간
- `1030`: 매도체결량 누계
- `1031`: 매수체결량 누계
- `1315`: 매도 단건 체결량 보조
- `1316`: 매수 단건 체결량 보조

왼쪽 목록 반영:

- 현재가
- 누적거래량
- 등락률
- 가격 색상

최근 체결 반영:

- 체결가격
- 체결량
- 시장가 매수 체결량은 빨강
- 시장가 매도 체결량은 파랑

체결량 누계:

- `1030/1031` 누계가 있으면 누계 기준으로 표시한다.
- 누계가 없으면 `1315/1316` 또는 `15`의 부호와 직전 틱 가격으로 보조 판단한다.
- 장마감 후 REST 체결 합계가 신뢰되지 않으면 매수/매도체결량은 공란 처리한다.

차트 반영:

- 선택 종목의 0B가 들어오면 현재 차트의 마지막 봉을 갱신한다.
- 일봉/주봉/月봉은 현재 날짜, 주, 월 버킷 기준으로 마지막 봉을 갱신한다.
- 분봉은 `20` 체결시간을 분 단위 버킷으로 묶어 마지막 봉을 갱신한다.
- 부분 확대 중에도 현재가 가로선과 축 라벨은 0B 현재가를 따라 움직인다.

## 0D/0H 호가

등록 대상:

- 현재 선택 종목 1개
- NXT 사용 중이면 `{종목코드}_NX`
- 아니면 KRX 6자리 코드

등록:

```json
{
  "trnm": "REG",
  "grp_no": "901",
  "refresh": "1",
  "data": [
    {
      "item": ["005930"],
      "type": ["0D", "0H"]
    }
  ]
}
```

0D 주요 필드:

- 매도호가: `41`~`50`
- 매도잔량: `61`~`70`
- 매수호가: `51`~`60`
- 매수잔량: `71`~`80`

0H 주요 필드:

- `10`: 현재가
- `12`: 등락률

호가 화면 규칙:

- 20호가 구조다.
- 가격은 가운데 가격 줄에 이어서 표시한다.
- 매도호가는 위쪽, 매수호가는 아래쪽이다.
- 매도잔량은 오른쪽 정렬한다.
- 매수잔량은 왼쪽 정렬한다.
- 중앙 현재가 박스는 제거했다.
- 모든 호가 가격 옆에 기준가 대비 등락률을 표시한다.
- 등락률은 소수 둘째 자리까지만 자르고 반올림하지 않는다.
- 가격 색상과 등락률 색상은 KRX 전일종가 기준이다.
- 현재가와 같은 호가 가격 칸은 어두운 청록 배경과 하늘색 1px 테두리로 표시한다.
- 현재가 강조 칸은 글자 흔들림을 막기 위해 가운데 호가 컬럼 폭을 102px로 유지한다.

장마감 후 보강:

- KRX 15:30 이후 KRX 종목은 REST 최종 스냅샷으로 호가와 현재가를 채운다.
- NXT 가능 종목은 NXT 20:00 최종 상황을 다음날 07:00 전까지 고정값처럼 본다.

REST 스냅샷:

- `ka10004`: 주식호가
- `ka10007`: 시세표성정보

## 상단 상태표시줄

화면 위치:

- 차트 상단의 얇은 상태줄

표시 항목:

- 종목명
- 거래량
- 거래대금
- 시가총액
- 유통주식수
- 회전율
- 주가등락률
- 전일거래량대비 금일 거래량 비율

주요 TR:

- `ka10100`: 종목 기본정보, 상장/유통 관련 보조
- `ka10007`: 현재가, 등락률, 거래량, 거래대금, 기준가 후보
- `ka10095`: 관심종목정보, 시가총액/거래량/거래대금 보조
- `ka10015`: 일별거래상세, 전일거래량대비 금일거래량 비율

전일거래량대비 금일거래량 비율:

- 금일 0봉 거래량 / 직전 거래일 1봉 거래량 * 100
- 오늘 거래량이 전일보다 많으면 빨강
- 적으면 파랑
- 보합 또는 데이터 부족은 흰색/공란

9. 매수비율
10. 프로그램

## 차트

화면 위치:

- 중앙 메인 차트
- 하단 거래량 막대 차트

기간:

- 일봉: 120봉
- 주봉: 100봉
- 월봉: 60봉
- 분봉: 700봉

분봉 종류:

- 1분
- 3분
- 5분
- 10분
- 15분
- 30분
- 60분
- 120분

사용 TR:

- 일봉: `ka10081`
- 주봉: `ka10082`
- 월봉: `ka10083`
- 분봉: `ka10080`

참고:

- SOR ON 상태에서 `_AL` 코드로 일/주/月/연 차트를 다운로드하는 예시는 `Docs/KiwoomReferences/sor-chart-download-reference.md`에 레퍼런스로 보관한다.
- 현재 화면 차트의 기본 정책은 KRX/NXT 분리이며, SOR 통합 차트는 별도 검증 후 적용한다.

분봉 요청:

- endpoint: `/api/dostk/chart`
- `tic_scope = 분 단위`
- `upd_stkpc_tp = "1"`
- `base_dt = 오늘`

NXT/KRX:

- KRX 차트는 6자리 종목코드로 요청한다.
- NXT 차트는 `_NX` 코드를 사용한다.
- 차트 캐시는 KRX/NXT 구분을 포함한다.
- KRX/NXT 차트 캐시는 서로 대체하지 않는다.
- 일/주/月 파일 캐시도 같은 구분을 따른다.

이평선:

- 5: 핑크
- 10: 블루
- 20: 노랑
- 60: 그린
- 분봉 추가:
  - 240
  - 480

부분 확대:

- 마우스 드래그로 구간 선택
- 더블클릭으로 전체 복귀
- 휠 확대/축소는 사용하지 않는다.
- 추가 과거 조회는 아직 하지 않는다.

부분 확대 시 이평선:

- 보이는 구간만으로 이평선을 다시 계산하지 않는다.
- 전체 로딩된 봉 기준으로 이평선을 계산하고, 화면에는 선택 구간만 그린다.
- 그래서 60/240/480선이 확대 구간 초반에서 불필요하게 잘리지 않아야 한다.

실시간 봉 갱신:

- 0B 현재가로 마지막 봉을 갱신한다.
- 같은 버킷이면 마지막 봉의 종가/고가/저가/거래량만 수정한다.
- 새 버킷이면 새 봉을 추가한다.
- 마지막 봉만 갱신 가능한 경우 전체 차트 재그리기를 피한다.
- 축 범위가 바뀌어야 하거나 새 봉이 생기면 전체 재그리기를 수행한다.

성능 로그:

- 예: `차트 전체그리기(3분봉 / 캐시): 700봉 / 160ms`
- 이 시간은 화면 그리기 시간이다.
- REST 데이터 조회 시간은 포함하지 않는다.

## 뉴스

화면 위치:

- 오른쪽 하단 `종목 뉴스 클리핑`

데이터:

- Naver News API

설정:

- `NaverNews.ClientId`
- `NaverNews.ClientSecret`
- `Dashboard.NewsCount`

처리:

- 종목 선택 시 종목명으로 검색한다.
- 선택 버전이 바뀌면 이전 뉴스 결과는 화면에 반영하지 않는다.
- 네이버 뉴스 API는 1초 5회 전역 제한을 둔다.
- 429 수신 시 `Retry-After`를 우선 사용하고, 없으면 1초 대기 후 1회만 재시도한다.
- 단일 뉴스 검색 요청 수량은 최대 100개로 제한한다.

조건식 편입 뉴스 알림:

- 파일: `MainWindow.LateNews.cs`
- 조건식 편입 종목이 들어오면 종목명으로 뉴스 후보 20개를 조회한다.
- 조회 횟수는 1회이며, 후보 개수만 20개로 넓힌다.
- `Config/NewsKeywords/news_keywords.json` 사전으로 제목/요약을 분석한다.
- 당일 뉴스는 소폭 우선점수를 준다.
- 점수 기준을 통과한 뉴스 중 상위 `LateNewsAlert.NewsCount`개만 텔레그램으로 보낸다.
- 기본 전송 개수는 3개다.
- 사전 단어가 하나도 잡히지 않거나 기준 점수를 통과하지 못하면 전송하지 않는다.
- 알림 통과 점수는 `positive` 계열 키워드 중심으로 계산한다.
- `sentiment = "positive"`로 명시된 카테고리만 알림 점수 가산 대상이다.
- `negative`/`caution` 계열 키워드는 점수 가산 재료로 쓰지 않고 태그 유지와 소폭 감점으로만 반영한다.
- 긍정 키워드와 함께 등장한 부정/주의 키워드는 알림을 무조건 차단하지 않고 보조 경고 정보로 함께 표시한다.
- 텔레그램 자동 뉴스 알림은 한 종목당 하나의 메시지로 묶어 보낸다.
- 메시지 상단에 `-종목명-`을 표시한다.
- 메시지 안에는 필터 통과 뉴스의 제목 전체를 먼저 표시하고, 바로 아래에 `기사보기 1`, `기사보기 2`, `기사보기 3` 링크를 넣는다.
- 첫 번째 링크는 보이지 않는 HTML 링크로 한 번 더 넣어 웹페이지 프리뷰가 1등 뉴스 기준으로 뜨게 한다.
- 텔레그램 전송은 전역으로 순차 처리하며 1.1초에 1회만 보낸다.
- 텔레그램 429 수신 시 `retry_after`를 우선 사용하고, 없으면 2초 대기 후 1회만 재시도한다.
- 이 필터는 매수 신호가 아니라, 필요 없는 뉴스를 버리는 뉴스 해석 보조 장치다.
- 키워드 사전이 없거나 로드 실패하면 텔레그램 뉴스 알림은 보내지 않고 로그만 남긴다. 화면 뉴스 목록은 별도 표시 흐름이므로 유지한다.
- 뉴스 필터 실패나 뉴스 제한은 키움 시세/호가/체결 흐름을 막지 않는다.

뉴스 시점 차트 이미지 확장 계획:

- 뉴스 알림에 15분봉 스크린샷을 붙일 수 있다.
- 화면을 직접 캡처하지 않고, 15분봉 데이터를 PNG로 따로 렌더링해서 텔레그램 `sendPhoto`로 보내는 방식을 우선한다.
- 네이버 뉴스 `PubDate`를 기준으로 해당 뉴스 시점 근처의 15분봉을 찾는다.
- 최근 40~80개 15분봉을 그린 뒤 뉴스 시점 봉에 세로선이나 마커를 표시한다.
- 현재 사용자가 보고 있는 종목/탭/봉 종류와 무관하게 정확한 이미지를 보내기 위해 UI 캡처 방식은 기본안에서 제외한다.

## 공시

화면 위치:

- 오른쪽 하단 공시 목록

데이터:

- DART API

설정:

- `Dart.ApiKey`
- `Dart.DisplayCount`
- `Dart.LookbackDays`

처리:

- 종목 선택 시 종목코드 기준으로 조회한다.
- 선택 취소 토큰을 사용한다.
- 이전 종목 결과가 늦게 와도 현재 화면에 반영하지 않는다.

조건식 편입 공시 알림:

- 파일: `Services/DartDisclosureAlertService.cs`
- 조건식 실시간 편입이 트리거다.
- 편입 종목의 DART `corp_code`를 찾고 오늘/어제 공시만 조회한다.
- 최근 공시 후보는 최대 5개 조회하고, 알림은 최대 3개만 보낸다.
- 공시 필터는 매수후보 종목의 좋은 재료 확인용 보조 필터다.
- 전체 공시에서 위험 공시를 찾는 기능이 아니다.
- 이미 조건식/수급/거래대금/가격 움직임으로 매수후보가 된 종목에 대해서만 좋은 방향의 공시를 우선 추적한다.
- 우선 추적 공시 키워드:
  - 수주
  - 공급계약
  - 신규시설투자
  - 증설
  - 자사주취득
  - 자사주소각
  - 배당
  - 주주환원
  - 기업가치제고
  - 밸류업
  - 지분취득
  - 합병
  - 인수
- 부정/위험 공시는 메인 필터가 아니라 예외 경고 태그 후보로 둔다.
- 부정 공시 때문에 전체 구조를 위험 감시기로 만들지 않는다.
- 중복 방지는 DART `rcept_no` 기준으로 한다.
- 앱 실행 중 같은 접수번호 공시는 다시 보내지 않는다.
- 날짜가 바뀌면 중복 캐시를 비운다.
- 텔레그램 메시지는 `-종목명 공시-` 상단 표기 후 `공시보기 1..3` 링크를 붙인다.
- 공시 알림은 조건식 편입 처리에서 뉴스 알림보다 먼저 실행한다.
- 공시 조회/알림 실패는 키움 실시간 등록 흐름을 중단하지 않는다.
- DART 공시검색 API는 `list.json`을 사용한다. 공식 문서 기준으로 `corp_code`, `bgn_de`, `end_de`, `sort=date`, `sort_mth=desc`, `page_count`를 사용한다.
- 공식 문서상 응답의 `rcept_no`는 공시뷰어 링크에 사용하는 접수번호이며 중복 방지 키로 사용한다.
- DART 조회는 전역으로 순차 처리하며 1초에 1회만 보낸다.
- 공식 문서상 제한 초과는 status `020`으로 내려올 수 있으므로, `020` 또는 HTTP 429 수신 시 DART만 1분 쿨다운한다.
- DART 제한 로그는 `DART limit: ...` 형식으로 남긴다.
- 전체 공시 추적기로 확장할 때는 DART 일일 호출 카운터를 추가한다.

공시 추적 확장 방향:

- 1단계: 돈을 쫓는 구조
  - 조건식 편입 종목만 오늘/어제 공시를 확인한다.
  - 현재 구현된 안정형이다.
- 2단계: 주기적 전체 공시 레이더
  - DART 전체 최신 공시를 주기적으로 조회한다.
  - `rcept_no` 기준으로 신규 공시만 누적한다.
  - 위험 감시기가 아니라 좋은 재료 후보를 찾는 별도 탭/레이더로 둔다.
  - 키움/매매 흐름보다 낮은 우선순위의 별도 엔진으로 운용한다.

## 캐시 정책

관심종목 캐시:

- 파일: `Config/watchlist_stock_cache.json`
- 개발 중 자주 바뀌는 정상 파일이다.
- 신규 조건식 편입 종목이 있을 수 있으므로 과도하게 신경 쓰지 않는다.

차트 메모리 캐시:

- 키:
  - 종목코드
  - KRX/NXT 여부
  - 차트 기간
- 최대 엔트리: 200개
- 최대 봉 수: 100,000개
- LRU 방식으로 오래된 캐시를 제거한다.

차트 파일 캐시:

- 파일: `Config/chart_candle_cache.json`
- 저장소: `Services/ChartCandleCacheStore.cs`
- 모델: `Models/ChartCandleCacheDocument.cs`
- 대상:
  - 일봉 120개
  - 주봉 100개
  - 월봉 60개
- 파일 보관:
  - 일봉 최대 600개
  - 주봉 최대 300개
  - 월봉 최대 180개
- 제외:
  - 화면 차트 파일 캐시는 분봉을 저장하지 않는다. 전략분봉 seed 파일은 별도 저장소 `Storage/StrategyMinuteSeeds`를 사용한다.
- 초기 조건식 결과에서 기준봉 게이트를 통과한 종목만 대상으로 한다.
- 프로그램 구동 후 백그라운드에서 1회만 조회하고 파일에 저장한다.
- 시작 직후 REST 경쟁을 줄이기 위해 몇 초 대기한 뒤 프리로드를 시작한다.
- 저장 시 기존 파일을 통째로 덮지 않고 날짜 기준으로 병합한다.
- 같은 날짜 봉은 최신 REST 값으로 교체하고, 저장 파일에 없던 날짜는 추가한다.
- 따라서 마지막 저장 이후 새로 생긴 일/주/月 봉은 누적되고, 장중 현재 봉도 최신값으로 보정된다.
- 화면 차트 조회 순서:
  1. 메모리 캐시
  2. 파일 캐시
  3. REST 조회
- 파일 캐시가 있으면 먼저 빠르게 보여주고, 이후 REST 조회가 성공하면 메모리 캐시를 갱신한다.
- KRX/NXT 여부와 차트 기간은 파일 캐시 키에 포함한다. KRX와 NXT 차트 캐시는 서로 대체하지 않는다.
- 이 기능은 화면 편의용 보조 기능이다. 별로면 `StartInitialChartFileCachePreload` 호출과 `ChartCandleCacheStore` 사용부를 제거하면 된다.

조건검색 재편입 캐시:

- 한번 편입된 종목은 기본정보를 캐시에 저장한다.
- 이후 재편입 시 REST 재조회 없이 캐시로 목록을 복원한다.
- 복원 후 0B 등록만 다시 맞춘다.

## REST 호출 제한 정책

목적:

- 키움 REST 429 방지

현재 정책:

- 최대 5 calls/sec 기준
- 지속 모드 간격 200ms
- 동시 게이트 5
- 429 또는 제한 return_code 수신 시 cooldown

관련 구현:

- `WaitRestRateLimitAsync`
- `SendKiwoomRestAsync`

주의:

- 빠른 종목 연속 선택에서 이미 서버로 전송된 REST 요청은 취소해도 서버에서는 처리될 수 있다.
- 따라서 UI 반영은 선택 버전으로 막고, REST 요청 자체는 호출 제한을 지켜야 한다.

## 전략 슬롯 구조

전략은 화면 코드에 직접 넣지 않는다.

용어 기준:

- 용어가 헷갈릴 때는 `Docs/trading-dashboard-glossary.md`를 먼저 따른다.
- 특히 `기준봉`, `장중 기준봉`, `화면 일봉`, `판단 일봉`, `SOR`, `_NX`, `_AL`은 임의로 섞지 않는다.
- 기준봉 정의와 화면 목록 진입 게이트는 `Docs/base-candle-definition-and-workflow.md`를 따른다.

기준:

- 전략 탭 최상단의 빨간 실행 영역에는 `Engine Start`, `Live Orders`, 예산, 슬롯수를 둔다.
- `Engine Start`가 ON이면 전략 신호 엔진과 알림/기록이 작동하며 예산/슬롯수 입력창은 잠긴다.
- `Live Orders`가 ON이면 전략 신호를 기존 실전 주문 레이어로 넘긴다.
- `Live Orders`가 OFF이면 `Engine Start`가 ON이어도 실주문은 내지 않고 알림/기록만 남긴다.
- 현재 단계의 전략 슬롯/컨트롤 보드는 실주문 실행부 확정이 아니라 전략 진행 상태 표시판과 모의 실행 검증판으로 먼저 운용한다.
- `Paper Trading`은 빨간 실행 영역 밖의 노란 독립 셀로 둔다. `Live Orders`가 OFF일 때만 켤 수 있다.
- `Paper Trading`은 `Engine Start ON + Live Orders OFF` 조합에서 테스트 매매 로그를 남기는 자리다. `Live Orders`가 ON이면 꺼지고 비활성화된다.
- `Live Orders`를 끄면 알림만으로 신호 품질을 우선 판단한다.
- `Engine Start` ON 상태에서는 `Live Orders` 여부와 상관없이 전략 슬롯과 전략실 키를 변경할 수 없다.
- 이때 전략 슬롯/전략실 키를 누르면 기존 상태로 되돌리고 `띵띵` 경고 로그를 남긴다. 색상은 비활성색으로 흐리지 않는다.
- 전략 탭 최하단에는 `전략실 전광판`을 둔다.
- 전광판은 자동매매 ON/OFF, 예산, 슬롯수, 선택된 전략, 전략실 키 값을 녹색 형광 글씨로 표시한다.
- 오른쪽 `Strategy` 탭은 적용/미적용 선택 UI만 담당한다.
- 각 전략은 `Services/Strategies/{StrategyName}` 아래 독립 폴더와 클래스로 만든다.
- `MainWindow.StrategySlots.cs`가 토글 상태를 읽어 ON인 전략만 평가한다.
- OFF 슬롯은 평가 전에 제외한다.
- 전략 클래스는 신호 판단 결과만 반환하고, 주문 여부는 별도 주문/리스크 레이어가 판단한다.

초기 슬롯:

- Slot 1: SOR 10분 MA60 눌림 + 3분 20봉 돌파 공격형 / `SOR` / 기준가는 KRX 전일종가 고정
- Slot 2: SOR 15분 MA60 눌림 + 5분 20봉 돌파 안정형 / `SOR` / 기준가는 KRX 전일종가 고정
- Slot 3: SOR 10분 MA60 눌림 + 5분 20봉 돌파 중간형 / `SOR` / 기준가는 KRX 전일종가 고정
- Assist: 테마/공시/뉴스 보조 / `보조` / 단독 매수신호 없음

후순위 보관 전략:

- 10순위: 당일 1분 장대양봉 눌림 / 문서 `Docs/Strategies/intraday-one-minute-base-candle-pullback.md`
- 주 매매법이 아니므로 현재 전략 슬롯 UI와 registry에는 등록하지 않는다.
- 이 전략의 1분 장대양봉은 장중 기준봉이며, 기존 KRX 일봉 기준봉 `BaseCandle*`, `BaseHalfPrice`를 덮어쓰면 안 된다.
- 기본 전제는 당일 등락률 20% 이상, 당일 거래대금 1,000억 이상, 1분봉 거래대금 10억 이상, 1분봉 시가 대비 종가 +2% 이상 양봉이다.
- 매수 후보는 장중 기준봉 중심값 지지 후 `접근 -> 반응 -> 확인`을 통과한 경우에만 검토한다.

폐기 전략:

- 전일상한가 몸통 눌림 시가회복 / 기존 `조건00 MA 회복초입` 계열은 현재 전략 슬롯 UI와 registry에서 제거한다.
- 관련 클래스와 문서는 이력 확인용으로만 보관하며, 새 전략 판단이나 Progress 계산에 연결하지 않는다.

SOR 당일강세 눌림 돌파 계열:

- 안정형: 15분봉 60선 눌림 + 5분봉 20봉 신고가 / 현재 Slot 2 / 문서 `Docs/Strategies/sor-fifteen-minute-ma60-five-minute-breakout.md`
- 중간형: 10분봉 60선 눌림 + 5분봉 20봉 신고가 / 현재 Slot 3 / 문서 `Docs/Strategies/sor-ten-minute-ma60-five-minute-breakout-middle.md`
- 공격형: 10분봉 60선 눌림 + 3분봉 20봉 신고가 / 현재 Slot 1 / 문서 `Docs/Strategies/sor-ten-minute-ma60-three-minute-breakout-aggressive.md`
- 공격형은 빠른 대신 속임수 돌파가 많으므로 체결강도, 돌파 후 과열 제한, 10분 60선 재이탈 손절을 함께 검증한다.
- 폐기된 `조건00 MA 회복초입` 계열은 공격형/안정형 SOR 전략과 섞지 않는다.
- 백테스트에서는 신호 지연, 10분 60선 눌림의 실제성, 돌파 후 밀림, 거래대금 1,000억 충족 시점, 손절폭 적정성을 반드시 체크한다.

운영 원칙:

- 매수전략 구현 전 `Docs/Strategies/strategy-composition-and-execution-rules.md`를 먼저 따른다.
- 새 전략 추가 시 새 클래스를 먼저 만들고 슬롯에 연결한다.
- 체크 해제된 전략은 후보 계산, 알림, 주문 연결을 하지 않는다.
- 기존 완성 전략은 문서 해석 그대로 옮기고 임의 변형하지 않는다.
- 각 전략은 `StrategySlotDescriptor`에 시장 범위 배지를 반드시 정의한다.
- KRX 전용, NXT 전용, SOR, 통합, 보조 전략을 UI 배지와 코드 메타데이터 양쪽에 같이 기록한다.
- 각 전략은 `Docs/Strategies` 아래 설명서를 반드시 가진다.
- 전략 탭의 `설명` 버튼은 `StrategySlotDescriptor.Detail` 내용을 보여주고, 연결 문서 경로를 같이 표시한다.

## 전략 중복 정책

같은 종목이 여러 전략에 동시에 걸릴 수 있다.

전략 탭 상단에는 두 개의 토글 스위치를 둔다.
전략실 키는 전략 슬롯 아래에 배치하고, 반으로 나누지 않고 한 줄 카드 형태로 세로 배치한다.

- `겹침 매수`
  - OFF: 이미 다른 전략에서 매수/보유/주문 진행 중이면 추가 매수하지 않는다.
  - ON: 전략별 추가 매수를 허용한다.
- `겹침 알림`
  - OFF: 중복 전략 신호를 조용히 기록만 한다.
  - ON: 중복 전략 신호를 알림으로 보낸다.

기본값:

```text
겹침 매수 OFF
겹침 알림 ON
```

공통 모델:

- `StrategyDuplicatePolicy`
- `StrategyExecutionSettings`

실행 모드 우선순위:

```text
Engine Start ON + Live Orders ON = live orders enabled
```

즉:

```text
Engine Start ON + Live Orders OFF = alerts and records only
Engine Start OFF = no strategy execution
```

## 전략 진행상황 표시

전략은 단발 조건이 아니라 상태 흐름으로 추적한다.

진행상황 탭은 다음 값을 기준으로 표시한다.

- 전략 다중선택 필터
- 보유/미보유 다중선택 필터
- 선택 전략에 속한 종목 리스트
- 현재 단계 번호
- 전체 단계 수
- 진행률 %
- 레벨 텍스트
- 강도 %
- 강도 라벨
- 상태 키
- 단계별 완료/현재 위치

공통 모델:

- `StrategyProgressStep`
- `StrategyProgressSnapshot`

현재 화면 연결:

- `Progress` 탭의 상단 필터는 전략 슬롯과 보유/미보유 상태를 고른다.
- 하단 목록은 정적 샘플이 아니라 현재 선택 종목을 `EvaluateEnabledStrategySlots`로 평가한 결과를 표시한다.
- 종목 선택, 전략 슬롯 토글, 진행 필터 토글, 잔고 새로고침, 선택 종목 상태 조회 이후 다시 그린다.
- 보유/미보유 기준은 현재 잔고 목록의 종목코드와 선택 종목코드를 정규화해서 비교한다.
- `StrategyProgressCalculator.Build`가 전략별 단계 수를 공통 진행률로 변환한다.
- 매수 전 단계 수가 5개든 10개든 0~70% 안에서 자동 분배한다.
- 보유 종목은 70%에서 매도 추적 구간을 시작하고, 매도 단계는 70~100% 안에서 자동 분배한다.
- Progress의 분봉 데이터 단계는 `_currentChartCandles` 하나를 공용으로 보지 않고, 현재 선택 종목의 차트 메모리 캐시에 들어온 분봉별 봉 개수로 확인한다.
- Slot 1은 10분/3분, Slot 2는 15분/5분, Slot 3은 10분/5분 데이터가 각각 최소 개수 이상 들어와야 해당 데이터 단계가 통과된다.
- 차트가 새로 로드되면 Progress를 다시 그려 방금 받아온 분봉 캐시 개수를 화면에 반영한다.
- 단, 현재는 매도 추적 로직을 아직 연결하지 않았으므로 실제 보유 종목은 중복매수 정책을 먼저 탄다.
- 보유 종목이 자동매수 편입 대상이 되면 `중복매수 OFF`에서는 제외하고, `중복매수 ON`에서는 다음 전략 처리로 넘긴다.
- 사용자가 임의로 보유 종목을 특정 전략 룰에 태우는 작업은 이후 수동 편입 기능으로 분리한다.

예시:

```text
Level 2 · 60% · RECOVERED_MA60
기준강도 70%
```

주의:

- 진행률은 매수 확률이 아니다.
- 진행률은 한 종목이 해당 전략 안에서 어디까지 진행됐는지를 보여주는 생애주기 표시다.
- 0~70%는 미보유 후보 추적 구간이다.
- 70%는 매수 체결 완료 기준선이다.
- 70~100%는 보유 이후 손절, 목표, 트레일링, 최종 매도 관리 구간이다.
- 강도 바는 해당 단계의 질을 표시한다. 예를 들어 기준봉 강도, 눌림 안정도, 회복 강도처럼 전략별로 의미가 다르다.
- 진행 탭은 상단에서 하나 이상의 전략 필터를 선택하고, 아래에는 선택된 전략들에 걸린 종목들을 단계별 진행바 형태로 보여준다.
- 상태 필터는 `미보유`와 `보유`를 독립 토글로 둔다.
- `미보유`와 `보유`가 둘 다 켜진 상태가 혼합 보기다. 별도 혼합 버튼은 만들지 않는다.
- 전략 OFF 상태에서는 진행률을 새로 계산하지 않는다.

생애주기 예시:

```text
0%   조건식 편입
15%  재료/뉴스/공시 확인
30%  기준봉 확인
45%  눌림 확인
60%  반응/돌파 대기
70%  매수 체결 완료
80%  손절가 상향 또는 1차 목표 접근
90%  1차익절/트레일링
100% 전량 매도/종료
```

## 개발 시 주의할 점

- 기준가를 KRX 전일종가 외 값으로 바꾸지 말 것.
- NXT 현재가와 KRX 기준가를 혼동하지 말 것.
- 조건검색식은 주기 리로드가 아니라 실시간 이벤트 기반이다.
- 편입/이탈했던 종목은 캐시로 처리하고 REST 재조회하지 않는 흐름을 유지할 것.
- 종목 선택 작업은 항상 선택 버전과 취소 토큰을 고려할 것.
- 빈 응답으로 기존 화면 값을 지우지 말 것.
- 새 화면 항목을 추가하면 이 문서에 데이터 출처, 계산 기준, 갱신 방식을 같이 추가할 것.
